 CORREÇÃO 1 — DEFEITO DE SINCRONIZAÇÃO NA SensorThread (CRÍTICO)
================================================================================

Arquivo: app/src/main/java/com/autotarget/service/SensorThread.java

PROBLEMA IDENTIFICADO PELO PROFESSOR:
  O método coletarDados() iterava a lista de alvos dentro de
  synchronized(sensorLock). Porém, a lista de alvos é protegida por
  collisionLock (não por sensorLock — são objetos diferentes). Se o
  spawnTimer adicionasse um alvo ou o physicsTimer removesse um alvo
  inativo durante a iteração da SensorThread, poderia ocorrer
  ConcurrentModificationException. Embora a janela de execução fosse
  estreita (SensorThread roda a cada 2s), era uma race condition latente
  e um defeito real de sincronização.

O QUE FOI FEITO:

  1. O construtor da SensorThread agora recebe o collisionLock como
     terceiro parâmetro:

     ANTES:
       public SensorThread(List<Alvo> alvos, Object sensorLock)

     DEPOIS:
       public SensorThread(List<Alvo> alvos, Object sensorLock, Object collisionLock)

  2. O método coletarDados() foi redesenhado em duas fases:

     Fase 1 — sob collisionLock:
       Adquire collisionLock, itera a lista de alvos com segurança,
       copia os valores de getX(), getY() e getVelocidade() de cada
       alvo ativo para arrays locais (localPosX, localPosY, localVel).
       Adiciona ruído gaussiano durante a cópia. Libera collisionLock
       ao sair do bloco synchronized.

     Fase 2 — sob sensorLock:
       Adquire sensorLock, publica os arrays locais nos campos
       compartilhados (leiturasPosX, leiturasPosY, leiturasVelocidade,
       quantidadeAlvosColetados), e chama sensorLock.notifyAll() para
       notificar a ReconciliacaoThread que há dados novos.

     ANTES:
       private void coletarDados() {
           synchronized (sensorLock) {
               // iterava alvos aqui — LOCK ERRADO
               for (Alvo a : alvos) { ... }
               sensorLock.notifyAll();
           }
       }

     DEPOIS:
       private void coletarDados() {
           float[] localPosX, localPosY, localVel;
           int localCount;

           synchronized (collisionLock) {
               // Fase 1: copia dados dos alvos sob o lock correto
               ...
           }

           synchronized (sensorLock) {
               // Fase 2: publica e notifica sob sensorLock
               leiturasPosX = localPosX;
               ...
               sensorLock.notifyAll();
           }
       }

  3. Lock ordering documentado no cabeçalho do arquivo:
       collisionLock (externo) → sensorLock (interno)
       Nunca adquirir na ordem inversa para evitar deadlock.

JUSTIFICATIVA TÉCNICA:
  A separação em duas fases garante que:
  - A iteração de alvos usa o lock correto (collisionLock), eliminando
    a possibilidade de ConcurrentModificationException.
  - O sensorLock é mantido apenas pelo tempo mínimo necessário (publicar
    arrays e notificar), reduzindo contenção com a ReconciliacaoThread.
  - A ordem de aquisição (collisionLock → sensorLock) é consistente em
    todo o projeto, prevenindo deadlock.

 CORREÇÃO 2 — ELIMINAÇÃO DA REDUNDÂNCIA DE DETECÇÃO DE COLISÃO
================================================================================

Arquivos modificados:
  - app/src/main/java/com/autotarget/engine/Jogo.java
  - app/src/main/java/com/autotarget/engine/GameSurfaceView.java

PROBLEMA IDENTIFICADO PELO PROFESSOR:
  A colisão era verificada em dois pontos diferentes:
  1. Na thread do Projetil via Projetil.verificarColisoes() — a cada
     16ms por cada projétil ativo, marca alvo + projétil como inativos.
  2. Na RenderThread da GameSurfaceView via jogo.verificarColisoes() —
     a cada frame (~33ms), contabiliza pontos e remove alvos inativos
     da lista.
  Isso concentrava contenção desnecessariamente no collisionLock e
  misturava lógica de negócio com renderização.

O QUE FOI FEITO:

  Em Jogo.java:
    1. Adicionado um PhysicsTimer dedicado, executando a cada 16ms (~60Hz):

       physicsTimer = new Timer("PhysicsTimer", true);
       physicsTimer.scheduleAtFixedRate(new TimerTask() {
           @Override
           public void run() {
               if (estado == Estado.RODANDO) verificarColisoes();
           }
       }, 0, 16);

    2. O PhysicsTimer é agora o ÚNICO responsável por chamar
       verificarColisoes(), que contabiliza pontos e remove alvos
       inativos da lista.

    3. O physicsTimer é cancelado em pararTimers():

       ANTES:
         private void pararTimers() {
             if (spawnTimer != null) { spawnTimer.cancel(); ... }
             if (gameTimer != null) { gameTimer.cancel(); ... }
         }

       DEPOIS:
         private void pararTimers() {
             if (spawnTimer != null) { spawnTimer.cancel(); ... }
             if (gameTimer != null) { gameTimer.cancel(); ... }
             if (physicsTimer != null) { physicsTimer.cancel(); ... }
         }

    4. A SensorThread agora recebe collisionLock no construtor:

       ANTES: sensorThread = new SensorThread(alvos, sensorLock);
       DEPOIS: sensorThread = new SensorThread(alvos, sensorLock, collisionLock);

  Em GameSurfaceView.java:
    Removida a chamada jogo.verificarColisoes() do método desenhar():

    ANTES:
      // Verificar colisões a cada frame
      if (jogo.getEstado() == Jogo.Estado.RODANDO) {
          jogo.verificarColisoes();
      }

    DEPOIS:
      // REMOVIDA A CHAMADA jogo.verificarColisoes() DESTE MÉTODO DE DESENHO.
      // A verificação de colisões agora é responsabilidade exclusiva do
      // PhysicsTimer (a cada 16ms) no Jogo.java, separando completamente
      // lógica de negócio de renderização.

JUSTIFICATIVA TÉCNICA:
  A separação Render/Physics é uma decisão arquitetural fundamental:
  - A RenderThread (30 FPS) APENAS lê estado e desenha. Não modifica
    listas nem contabiliza pontos.
  - O PhysicsTimer (16ms / ~60Hz) APENAS processa colisões, conta
    pontos e remove alvos inativos.
  - Isso elimina contenção desnecessária no collisionLock e simplifica
    o modelo mental: um único responsável por cada preocupação.

 CORREÇÃO 3 — POLIMORFISMO EM VEZ DE instanceof NA RENDERIZAÇÃO
================================================================================

Arquivos modificados:
  - app/src/main/java/com/autotarget/model/Alvo.java
  - app/src/main/java/com/autotarget/model/AlvoComum.java
  - app/src/main/java/com/autotarget/model/AlvoRapido.java
  - app/src/main/java/com/autotarget/engine/GameSurfaceView.java

PROBLEMA IDENTIFICADO PELO PROFESSOR:
  Na GameSurfaceView, linha 268, a cor do alvo era determinada por:
    Paint paint = (alvo instanceof AlvoRapido) ? paintAlvoRapido : paintAlvoComum;
  Isso é um downcasting implícito — a View precisa conhecer as subclasses
  concretas. Adicionar um terceiro tipo de alvo na AV2 (ex: AlvoEvasivo)
  exigiria modificar a GameSurfaceView. Uma alternativa mais polimórfica
  seria um método na classe base que cada subclasse implementa.

O QUE FOI FEITO:

  Em Alvo.java:
    Adicionado método abstrato getCorId():

    /**
     * Retorna o identificador de cor (ARGB int) para renderização polimórfica.
     * Cada subclasse define sua própria cor, eliminando a necessidade de
     * instanceof na GameSurfaceView — design extensível para novos tipos de alvo.
     *
     * @return cor no formato 0xAARRGGBB
     */
    public abstract int getCorId();

  Em AlvoComum.java:
    Implementação retornando verde (#4CAF50):

    @Override
    public int getCorId() {
        return 0xFF4CAF50;
    }

  Em AlvoRapido.java:
    Implementação retornando laranja (#FF9800):

    @Override
    public int getCorId() {
        return 0xFFFF9800;
    }

  Em GameSurfaceView.java:
    1. Removido o import de AlvoRapido (não é mais necessário).

    2. Substituída a verificação instanceof:

       ANTES:
         Paint paint = (alvo instanceof AlvoRapido) ? paintAlvoRapido : paintAlvoComum;

       DEPOIS:
         Paint paint = paintForColor(alvo.getCorId());

    3. Adicionado método utilitário paintForColor():

       private Paint paintForColor(int color) {
           Paint paint = new Paint();
           paint.setColor(color);
           paint.setAntiAlias(true);
           paint.setStyle(Paint.Style.FILL);
           return paint;
       }

JUSTIFICATIVA TÉCNICA:
  Com getCorId() na classe abstrata Alvo, qualquer nova subclasse criada
  na AV2 (ex: AlvoEvasivo que foge de projéteis próximos) apenas precisa
  implementar getCorId() retornando sua cor. A GameSurfaceView não precisa
  ser modificada — design aberto para extensão, fechado para modificação
  (princípio Open/Closed). A View não conhece mais as subclasses concretas.

 CORREÇÃO 4 — CAPTURAS GENÉRICAS SILENCIOSAS
================================================================================

Arquivos modificados:
  - app/src/main/java/com/autotarget/service/ReconciliacaoThread.java
  - app/src/main/java/com/autotarget/engine/GameSurfaceView.java

PROBLEMA IDENTIFICADO PELO PROFESSOR:
  Em dois pontos críticos do código havia catch (Exception e) {} sem
  nenhum log — erros eram silenciados completamente. Em produção, isso
  esconde NullPointerException, IllegalStateException, etc, dificultando
  diagnóstico de problemas.

O QUE FOI FEITO:

  Em ReconciliacaoThread.executarReconciliacao() (1 ponto):

    ANTES:
      } catch (Exception e) {
          // Falha na reconciliação, continua
      }

    DEPOIS:
      } catch (Exception e) {
          // Falha na reconciliação — registra para debug, mas não interrompe o loop
          android.util.Log.e("ReconciliacaoThread", "Erro na reconciliação", e);
      }

  Em GameSurfaceView.RenderThread.run() (2 pontos):

    ANTES:
      } catch (Exception e) {
      } finally {
          if (canvas != null) {
              try { holder.unlockCanvasAndPost(canvas); } catch (Exception e) {}
          }
      }

    DEPOIS:
      } catch (Exception e) {
          Log.e("RenderThread", "Erro no render loop", e);
      } finally {
          if (canvas != null) {
              try { holder.unlockCanvasAndPost(canvas); } catch (Exception e) {
                  Log.e("RenderThread", "Erro ao postar canvas", e);
              }
          }
      }

JUSTIFICATIVA TÉCNICA:
  Adicionar Log.e() mantém o comportamento desejado (a thread não quebra),
  mas registra a falha no Logcat para debug futuro. Qualquer exceção
  inesperada que ocorra durante renderização ou reconciliação agora será
  visível nos logs, permitindo diagnóstico sem precisar reproduzir o
  cenário exato.

 CORREÇÃO 5 — DEFENSIVE COPY NOS GETTERS DE LISTAS INTERNAS
================================================================================

Arquivo: app/src/main/java/com/autotarget/engine/Jogo.java

PROBLEMA IDENTIFICADO PELO PROFESSOR:
  getAlvos() e getCanhoes() retornavam a referência interna diretamente
  (a CopyOnWriteArrayList). Embora a CopyOnWriteArrayList seja thread-safe
  para iteração, um consumidor descuidado poderia adicionar ou remover
  elementos da lista retornada, corrompendo o estado interno do Jogo.

O QUE FOI FEITO:

  ANTES:
    public List<Alvo> getAlvos() { return alvos; }
    public List<Canhao> getCanhoes() { return canhoes; }

  DEPOIS:
    /**
     * Retorna defensive copy da lista de alvos.
     * Protege a referência interna contra modificação externa indevida.
     */
    public List<Alvo> getAlvos() { return new ArrayList<>(alvos); }

    /**
     * Retorna defensive copy da lista de canhões.
     * Protege a referência interna contra modificação externa indevida.
     */
    public List<Canhao> getCanhoes() { return new ArrayList<>(canhoes); }

  Adicionado import necessário:
    import java.util.ArrayList;

JUSTIFICATIVA TÉCNICA:
  A defensive copy é o trade-off clássico de imutabilidade vs custo de
  alocação. Como getAlvos() e getCanhoes() são chamados principalmente
  pela GameSurfaceView a ~30 FPS, o custo de alocação é aceitável
  (ArrayList de referências, sem deep copy dos objetos). A proteção
  contra uso indevido futuro compensa o custo mínimo por chamada.
  A lista interna (CopyOnWriteArrayList) permanece intacta — operações
  como alvos.add() e alvos.remove() dentro do Jogo continuam funcionando
  normalmente.

 CORREÇÃO 6 — DOCUMENTAÇÃO DE LOCK ORDERING
================================================================================

Arquivos modificados:
  - app/src/main/java/com/autotarget/engine/Jogo.java
  - app/src/main/java/com/autotarget/service/SensorThread.java

PROBLEMA IDENTIFICADO PELO PROFESSOR:
  A ordem de aquisição de locks estava documentada apenas parcialmente
  (comentário na GameSurfaceView sobre canhoes → projeteis). A ausência
  de inversões salvava o sistema de deadlock, mas a regra global não
  estava documentada.

O QUE FOI FEITO:

  No cabeçalho de Jogo.java, adicionada seção explícita:

    LOCK ORDERING (regra global — nunca adquirir na ordem inversa):
      collisionLock → sensorLock
      collisionLock → projeteis (dentro de Projetil.verificarColisoes)
      canhoes (externo) → projeteis (interno, dentro de GameSurfaceView)

  No cabeçalho de SensorThread.java, mesma regra replicada:

    LOCK ORDERING (regra global do projeto):
      collisionLock → sensorLock (nunca na ordem inversa)
      collisionLock → projeteis (dentro de Projetil/Canhao)
      canhoes → projeteis (dentro de GameSurfaceView)

  No método coletarDados() da SensorThread, comentário inline:

    Lock ordering: collisionLock (externo) → sensorLock (interno).

JUSTIFICATIVA TÉCNICA:
  Documentar a regra global de lock ordering em todos os arquivos que
  adquirem locks múltiplos serve como salvaguarda para desenvolvimento
  futuro: qualquer desenvolvedor que precise adicionar novo código de
  sincronização pode consultar o cabeçalho para verificar a ordem
  correta de aquisição, prevenindo deadlocks.

 CORREÇÃO 7 — @After tearDown() NOS TESTES DE CANHÃO
================================================================================

Arquivo: app/src/test/java/com/autotarget/CanhaoTest.java

PROBLEMA IDENTIFICADO PELO PROFESSOR:
  Testes que chamam canhao.disparar() internamente criam threads de
  Projetil (projetil.start()). O teste tentava limpar manualmente via
  setAtivo(false) + interrupt() no final de cada teste, mas havia janela
  de escape — threads podiam vazar entre testes. Em projetos maiores,
  isso causa flaky tests.

O QUE FOI FEITO:

  1. Adicionado import:
       import org.junit.After;

  2. Adicionada lista para rastrear canhões criados:
       private final java.util.List<Canhao> canhoesTestados = new java.util.ArrayList<>();

  3. No @Before, a lista é limpa:
       canhoesTestados.clear();

  4. Adicionado método @After tearDown():

       @After
       public void tearDown() {
           for (Canhao c : canhoesTestados) {
               c.pararCanhao();  // desativa canhão + todos seus projéteis
               c.interrupt();
           }
           for (Alvo a : alvos) {
               a.setAtivo(false);
               a.interrupt();
           }
       }

  5. Cada teste agora registra o canhão criado:
       canhoesTestados.add(canhao);

  6. Removida a limpeza manual que existia apenas no testDisparoComAlvoMesmoLado:

       REMOVIDO:
         // Limpar
         for (Projetil p : canhao.getProjeteis()) {
             p.setAtivo(false); p.interrupt();
         }

     Essa limpeza agora é feita uniformemente pelo @After.

JUSTIFICATIVA TÉCNICA:
  O @After é executado SEMPRE após cada teste (mesmo que o teste falhe
  com exceção), garantindo que todas as threads criadas sejam desativadas
  e interrompidas. Isso é especialmente importante porque:
  - pararCanhao() desativa o canhão E itera todos os seus projéteis,
    chamando setAtivo(false) + interrupt() em cada um.
  - A limpeza é centralizada em um único lugar em vez de duplicada em
    cada teste individual.
  - Previne flaky tests causados por threads de testes anteriores
    interferindo em testes subsequentes.
